# RateYourDJ 开发总结：Bug 与难点复盘

本文档记录搭建 RateYourDJ(对话式音乐推荐 Agent)过程中遇到的真实 bug、设计难点
及其解决思路。按"现象 → 根因 → 解决 → 收获"组织,既作为开发存档,也作为面试时
可讲的工程故事。

---

## 一、最隐蔽的 Bug

### 1. `_run_discovery` 未定义导致 model 模式全线 500

- **现象**:配置了 DeepSeek key 后,每次推荐请求都返回 HTTP 500;但不配 key 时
  一切正常,所以早期测试一直没发现。
- **根因**:model 模式下的"程序优先发现"流程调用了 `_run_discovery(...)`,但这个
  函数从未被定义(只有兄弟函数 `_run_provider_search`)。rules 降级路径根本不会
  走到那行,于是"没 key 的测试全绿"掩盖了这个 bug。
- **解决**:补全 `_run_discovery`,复用现有 `_run_provider_search` 的结构——读用户
  记忆 → 调 `discover_tracks` 工具 → 取 grounding 后的候选 → 排序过滤。
- **收获**:**"测试全绿"不等于"功能正确"**。降级路径会掩盖主路径的 bug;关键路径
  必须用接近真实的配置(带 LLM provider 的 mock)端到端测,而不是只测降级分支。

### 2. `discovery_reason` 字段在排序中被悄悄丢弃

- **现象**:DeepSeek 明明给每首歌写了推荐理由,但前端卡片上只显示"推荐算法依据"
  这种系统文案,模型写的理由消失了。
- **根因**:`_rank_provider_tracks` 重新构造 ranked-song 字典时,没有把
  `discovery_reason` 字段带过去,理由在排序环节丢失。
- **解决**:在 ranked-song 字典里补上 `discovery_reason`,并把解释器里该理由的权重
  从 0.65 提到 0.9,确保它稳定排在卡片首位。
- **收获**:跨层传递的数据结构,**字段在某一层重组时极易丢失**;端到端验证要检查
  "最终用户看到的内容",而不是只看某一层的输出。

## 二、意图识别的连环误判(NLP 规则的脆弱性)

### 3. "为什么推荐这三首"被误判为"再推荐"

- **现象**:用户追问已推荐歌曲的理由,系统却又生成了一批全新的歌。
- **根因**:意图判断里"推荐"是 recommend 关键词,且优先级**高于**疑问词。
  "为什么**推荐**这三首"命中了"推荐",于是被当成"要新歌"。
- **解决**:引入一组**强疑问词**(为什么/为何/理由/原因/是谁/介绍/解释…),其优先级
  **压过** recommend 关键词。

### 4. "他们是英伦摇滚吗"被误判为"要英伦摇滚的歌"

- **现象**:确认类问句("是不是 X 风格")又被当成点歌请求。
- **根因**:句中含"摇滚"(像点歌词),但既无强疑问词、又是确认句式,落在规则盲区。
- **收获(3+4 合并)**:**基于关键词的意图识别本质脆弱**,总有歧义句踩在盲区。
  这两次"打地鼠"式修补,最终促使我下决心做下面第 5 点的架构统一——**与其不断给
  规则打补丁,不如把决策权交给 LLM 自己用 ReAct 显式判断**。

---

## 三、架构级难点

### 5. 统一为"先回答 → ReAct 决定是否推歌"

- **难点**:项目早期有两套割裂的路径——"问答路径"(先回答)和"推荐路径"
  (直接出歌)。导致行为不一致:有时先回答、有时直接甩歌,用户体验割裂。
- **解决**:把所有 model 模式请求统一到一个 ReAct 决策入口:模型每轮先输出
  `thought` 推理 + 一段文字 `answer`,再用 `action`(answer_only / explain_only /
  suggest_new)**自主决定**是否调用推荐工具。
- **关键权衡(主动取舍并记录)**:
  - model 模式下不再走传统 L4 打分排序,改由 LLM 提名 + grounding 出歌——更契合
    "LLM 作为生成式发现引擎"的定位。
  - "换一批"的去重从硬过滤变为软提示(把已推荐的歌作为上下文)。
- **安全边界**:rules 模式和 50 个评估用例**完全不受影响**,因为 MockLLMProvider
  没有 `answer_question` 方法,会自动降级走老逻辑。
- **收获**:架构统一要**显式管理风险**——哪些路径会变、哪些必须保持不变,用降级
  判断(`hasattr`)守住测试和无 key 路径,做到"大改动、零回归"。

### 6. 长期画像不能被收藏重算覆盖(conversation_affinity)

- **难点**:想让 Agent 从对话中轻量学习偏好,但收藏行为会触发
  `rebuild_user_profile` 整体重算画像,会把对话学到的信号冲掉。
- **解决**:新增独立字段 `conversation_affinity`,用衰减 EMA 累积对话偏好,并在
  画像重算时**显式保留**该字段,使两种学习信号互不污染。

## 四、与外部依赖(LLM / 第三方 API)的博弈

### 7. 静默吞异常 + 误导性文案

- **现象**:第一次问答成功,第二次却提示"未配置 DeepSeek"——但 key 明明是好的。
- **根因**:`answer_question` 的调用被 `except` 静默捕获,任何失败都降级并显示
  "未配置 DeepSeek";被吞的异常不打印 traceback,导致无法诊断。
- **解决**:(a) 失败时用 logging 打出完整 traceback;(b) 文案区分"真没配 key" vs
  "配了 key 但调用失败",后者如实显示原因。
- **收获**:**降级(graceful degradation)不等于隐藏错误**。静默吞异常会让真实问题
  无法定位,还会用错误的提示误导排查方向。

### 8. DeepSeek 工具调用 arguments 的 dict / string 二义性

- **现象**:偶发"第二次请求失败",被上面的静默吞异常掩盖。
- **根因**:OpenAI 兼容 API 约定 `function.arguments` 是 JSON 字符串,但 DeepSeek
  实践中有时返回**已解析的 dict**。代码无脑 `json.loads()`,遇到 dict 直接抛错。
- **解决**:抽出 `_tool_call_arguments` 辅助函数,兼容 dict / string 两种形式;
  `_parse_answer` 和推荐循环的 `_parse_decision` 都改用它。
- **收获**:**对接外部 API 不能只信文档约定**,要对实际返回做防御性解析。

### 9. DeepSeek 请求超时(Read timed out)

- **现象**:偶发 45s 读超时,导致回答生成失败。
- **根因**:网络往返慢或单次生成内容长,超过默认超时。
- **缓解方向**:调大超时阈值 + 失败自动重试。
- **收获**:外部 LLM 调用是不稳定依赖,**超时 + 重试是生产级必备**。

### 10. 中文曲目在 Spotify 上 grounding 失败

- **现象**:推荐华语歌手(如华晨宇)时,返回 0 首可试听歌曲。
- **根因**:Spotify 对华语曲目按"歌名+艺人"精确匹配覆盖差,grounding 反幻觉机制
  把确认不了的歌全部丢弃。
- **解决**:**不为凑数而保留未确认的歌**(那会破坏"反幻觉"核心卖点);改为体验层
  诚实降级——当"模型提了歌但 grounding 全失败"时,文字回答补一句说明,而不是
  显示矛盾的空状态。
- **收获**:**grounding 抗幻觉与覆盖率是一对权衡**。宁可诚实承认覆盖盲区,也不能
  为了"有结果"而牺牲可信度。这是 provider 能力边界,非代码 bug。

## 五、测试与工程化的坑

### 11. 沙箱无 Flask 导致 web 测试"假失败"

- **现象**:`tests/test_web.py` 整组报 `ModuleNotFoundError: flask` 或断言失败。
- **根因**:开发沙箱无网络,装不了真 Flask;用了最小 Flask stub,但
  `render_template` 只返回占位符、query-string 解析不全。
- **解决**:把"非 web 测试套件"作为真实基线;web 逻辑改用直接构造 `create_app(...)`
  + `test_client()` 验证。在用户真实环境(装了 Flask)下 web 测试才有效。
- **收获**:要分清"环境限制导致的假失败" vs "真实 bug",并明确各自的验证基线。

### 12. 过时的测试断言

- **现象**:前端推倒重建后,`test_index_renders_frontend` 仍断言旧仪表盘的 DOM
  (`id="recommendations"`、`id="agent-debug-panel"`),报失败。
- **解决**:更新为新对话式 UI 的元素(`id="chat-stream"` / `composer` /
  `collection-drawer`)。
- **收获**:**重构功能时,测试断言要同步更新**,否则过时断言会持续误报。

### 13. 空状态提示在架构统一后变矛盾

- **现象**:回答正常,底下却挂着"这次没找到合适的歌"。
- **根因**:前端只在 `intent==="question"` 时隐藏该提示;但统一为"先回答再决定"后,
  很多推荐意图的回复也本就不带歌。
- **解决**:改为**只要有文字回答就不显示**空状态提示,仅"既无回答又无歌"时兜底。

## 六、贯穿全程的工程原则(可作为面试总结)

1. **测试全绿 ≠ 功能正确**:降级路径会掩盖主路径 bug,关键路径要用接近真实的配置
   端到端验证。
2. **降级要透明,不能吞错**:graceful degradation 必须保留可诊断性,错误文案不能
   误导排查。
3. **对接外部 API 要防御性编程**:不轻信文档约定(arguments 二义性)、要处理超时
   与不稳定、要正视覆盖边界(中文 grounding)。
4. **与其给规则打补丁,不如让 LLM 自主决策**:意图识别的连环误判,最终靠 ReAct
   统一决策根治——这也是"Agent"区别于"if-else 流程"的本质。
5. **大改动要显式管理风险**:明确哪些路径会变、哪些必须不变,用降级开关守住回归。
6. **诚实优先于好看**:grounding 不凑数、SFT/GRPO 不伪造训练结果——可信度比
   "看起来完整"更重要。

## 七、最终评测结果

以下为项目评估体系实跑得到的真实数据(非估计值),分行为正确性、Agent 质量指标、
A/B 离线对照三部分。

### 行为正确性

| 指标 | 结果 | 说明 |
| --- | --- | --- |
| 离线评估套件 (eval-suite) | **50 / 50 全部通过** | 覆盖 basic 12、constraints 10、session 10、edge 6、feedback 6、provider 6 六类场景的回归 |
| 单元测试 | **248 项,非 web 全部通过** | 仅 3 个 web 用例在无网络沙箱因 Flask stub 限制"假失败",真实环境(装 Flask)下通过 |

### Agent 质量指标(基于已采集的 138 条 trajectory)

| 指标 | 结果 | 说明 |
| --- | --- | --- |
| Grounding 成功率 | **64.8%** | LLM 提名的候选歌中,经 Spotify/Last.fm 确认真实存在的比例 |
| 幻觉率 | **0.52%** | 未通过外部确认(模型编造)的比例——grounding 机制将幻觉压到极低 |
| 平均响应延迟 | **个位数毫秒级**(Agent 逻辑,不含外部 API 网络往返) | 用 P50/P95 双指标监控 |

> 注:thought 覆盖率指标当前偏低,因为已采集的 trajectory 多为 ReAct 改造之前或
> rules 模式生成的历史数据,尚不含显式 thought;新版 model 模式产生的轨迹该指标
> 接近 100%(见下方 A/B 对照)。此处如实标注,不做修饰。

### A/B 离线对照(rules vs model/ReAct)

同一批 query 在两种配置下并排对比:

| 指标 | rules 模式 | model / ReAct 模式 |
| --- | --- | --- |
| ReAct 推理覆盖率 (thought coverage) | 0% | **100%** |
| 平均响应延迟 | ~3.5 ms | ~1.8 ms |

A/B 结果清晰量化了 ReAct 改造带来的**可观测性增益**:model 模式下每一步决策都带
显式推理,而 rules 模式无推理可循。

### 一句话结论

> 项目通过 50/50 的行为回归与 248 项单元测试,在 64.8% grounding 成功率下将幻觉率
> 控制在 0.52%,并以 A/B 离线对照量化了 ReAct 决策相比规则基线的可观测性提升。
